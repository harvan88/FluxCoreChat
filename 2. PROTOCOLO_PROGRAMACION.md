## 🎯 Patrón de Transición de Hitos Estructurado (HTP - Hito Transition Pattern)

### Filosofía
> "Nada se da por terminado hasta que esté verificado, documentado y probado."

---

## 📐 Estructura del Patrón (Template Universal)

### PARA AVANZAR DEL HITO [X] AL HITO [Y]:

#### 1. CIERRE DEL HITO ACTUAL ([X]):

- ✅ Verifique que todas las pruebas definidas para [X] hayan sido ejecutadas y aprobadas.
- ✅ Confirme el cumplimiento total de sus criterios de aceptación.
- ✅ Documente cualquier deuda técnica o pendiente menor.
- ✅ Cree punto de restauración (commit/tag) si es necesario.

#### 2. INICIO DEL NUEVO HITO ([Y]):

- Una vez validado [X], proceda a ejecutar las tareas específicas del Hito [Y].
- Defina criterios de aceptación MEDIBLES antes de comenzar.
- Identifique dependencias y riesgos.

---

## ✅ CRITERIOS DE FINALIZACIÓN PARA CUALQUIER HITO

El hito se considerará completado cuando se cumpla **AL MENOS**:

| Criterio | Descripción |
|----------|-------------|
| ✅ **Implementación Funcional** | El código/feature/especificación está completo |
| ✅ **Pruebas Exitosas** | Se ejecutan y aprueban pruebas relevantes (unitarias, integración, funcionales) |
| ✅ **Errores Resueltos** | Los issues identificados durante el desarrollo están cerrados |
| ✅ **Documentación Actualizada** | Se registran cambios, decisiones y lecciones |
| ✅ **Guía de Verificación** | Existe una lista numerada para pruebas manuales (si aplica) |
| ✅ **Build Exitoso** | `bun run build` pasa sin errores |

## 🔄 Protocolo de Continuidad entre Sesiones

Para garantizar **continuidad del trabajo** cuando se terminan créditos o se cambia de sesión:

### Al FINALIZAR una sesión de trabajo:

1. **Documentar estado actual** en el archivo de hitos correspondiente
2. **Crear checkpoint** con commit descriptivo
3. **Dejar TODO claro** con siguiente paso concreto
4. **Actualizar** `1. EXECUTION_PLAN.md` con estado actual

### Al INICIAR una nueva sesión:

1. **Leer** `1. EXECUTION_PLAN.md` para contexto
2. **Verificar** último estado documentado
3. **Ejecutar** `bun run build` para confirmar estado
4. **Continuar** desde el último TODO documentado

---

## ⚠️ Reglas de Oro

1. **NO avanzar sin verificar** - Cada hito debe cumplir sus criterios antes de pasar al siguiente
2. **Documentar SIEMPRE** - Si no está documentado, no existió
3. **Build primero** - Antes de cualquier acción, verificar que el build pasa
4. **Commits atómicos** - Un commit = una unidad de trabajo coherente
5. **Comunicar riesgos** - Si hay bloqueos o incertidumbre, documentar y consultar
